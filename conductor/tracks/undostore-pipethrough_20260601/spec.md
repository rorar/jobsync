## Summary

Today only `dismiss` is reversible from the staging deck. Code-verified at HEAD `b4b20e9`:
`REVERSIBLE_DECK_ACTIONS = ["dismiss"]` (`src/hooks/useDeckStack.ts:35`). `promote`,
`superlike`, and `block` produce server-side effects (a Job, a block row) but cannot be
undone from the deck because the server's compensation token never reaches the client
undo path — the `undoTokenId` field on the undo entry is **dead code today**
(`useDeckStack.ts:59`: "effectively dead code today"). This track pipes the token end-to-end
so those three actions can safely join the reversible allowlist.

## Context — the split-brain

The deck has a state machine (`useDeckStack`) and a server-side compensation store
(`src/lib/undo/undo-store.ts`, `compensate()` at `:84`). They are not connected for
Job-producing actions:
- **Server side:** actions return an `ActionResult`; the compensation lives in `undoStore`
  keyed by a token.
- **Client side:** `useDeckStack.onAction` returns `{ success, createdJobId }` (ADR-030
  Decision A) but carries **no** `undoTokenId`. `handleDeckUndo`
  (`src/components/staging/StagingContainer.tsx:503-532`) can only locally reverse the
  state-machine move; it has a drift-guard (`:532`) that logs if a non-reversible action
  ever reaches it. So promote/superlike/block stay OUT of `REVERSIBLE_DECK_ACTIONS`.

The fix is to thread the token: `ActionResult.data.undoTokenId` →
`useDeckStack.onAction` return → `UndoEntry` (`src/lib/undo/types.ts`) → `handleDeckUndo`
→ `undoStore.compensate(tokenId)`.

## Acceptance Criteria

- [ ] **ADR-030 amendment** recorded: `onAction` returns

- [ ] The server actions for `promote` / `superlike` / `block` register a compensation in
      `undoStore` and return its `undoTokenId` in `ActionResult.data`.
- [ ] `useDeckStack` forwards `undoTokenId` into the recorded `UndoEntry` (the field stops
      being dead code).
- [ ] `handleDeckUndo` calls `undoStore.compensate(tokenId)` for token-bearing entries;
      `promote`, `superlike`, `block` are added to `REVERSIBLE_DECK_ACTIONS`.
- [ ] The drift-guard at `StagingContainer.tsx:532` is updated/removed consistently with
      the widened allowlist.
- [ ] Undoing a super-like also removes the celebration (`onSuperLikeUndone` path,
      ADR-030) and compensates the server effect (no orphan Job).
- [ ] Tests: unit (token threading through `useDeckStack`), component (undo a promote/
      superlike/block reverses both client state AND server effect), regression
      (`{ success }`-only callers unaffected), ≥1 E2E happy-path (superlike → undo → Job gone).
