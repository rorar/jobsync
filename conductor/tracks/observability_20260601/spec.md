## ⛔ Entry Criteria

**Do not begin implementation until:** a stack decision is recorded (ADR) choosing
between **(a)** self-hosted OpenTelemetry + Tempo (tracing) + Prometheus (metrics) +
Grafana (dashboards), or **(b)** an off-the-shelf APM (e.g. a hosted vendor). This is a
2–3 week dedicated sprint and an architectural commitment — it must not be started as
part of a cleanup pass. Until the ADR lands, this track stays `pending` and untouched.

## Summary

JobSync today has **zero** structured observability: no OpenTelemetry, no Prometheus
metrics, no distributed tracing, no Core Web Vitals capture. Code-verified at HEAD
`b4b20e9`: a grep for `OpenTelemetry` / `@opentelemetry` / `prom-client` / `prometheus`
across `src/` + `package.json` returns nothing. The app logs ad-hoc to stderr
(`[admin-audit]`, `[emitEvent]`, `[EnrichmentTrigger]` console lines) but nothing is
aggregated, traced, or alertable. This track adds production-grade observability once the
stack is chosen.

## Context

The project already has natural instrumentation seams that the chosen stack should hook:
- **Domain events:** `TypedEventBus` (`src/lib/events/event-bus.ts`) — 29 event types,
  an obvious span/metric source.
- **Scheduler:** `RunCoordinator` (`src/lib/scheduler/run-coordinator.ts`) reports phase
  progress (search → dedup → enrich → match → save → finalize) — natural trace spans.
- **Connector resilience:** Cockatiel policies (`resilience.ts`) emit circuit-breaker /
  retry events — natural metrics.
- **Notification dispatch / API routes / server actions** — request-level spans + RED metrics.

(Instrumentation seams are graph-plausible navigation hints; **verify each seam against
the current file before wiring** — they were code-spotted, not graph-trusted.)

## Acceptance Criteria (to be refined after the stack ADR)

- [ ] **Tracing:** distributed traces across server actions → Prisma → external Connector
      fetches, with the scheduler run lifecycle as a parent span.
- [ ] **Metrics:** RED (rate/errors/duration) for API + server actions; connector
      circuit-breaker + rate-limiter gauges; scheduler run counts/durations.
- [ ] **Core Web Vitals:** client-side LCP/INP/CLS capture reported to the backend.
- [ ] **Dashboards + alerts:** Grafana (or APM) dashboards for the golden signals; alert
      rules on error-budget burn.
- [ ] **SLI/SLO:** at least one SLO defined (e.g. automation-run success rate) with an
      error budget.
- [ ] Tests: unit for the instrumentation wrappers; an integration test asserting a span
      is emitted for a representative server action.
- [ ] No PII in spans/metrics labels (GDPR — reuse `@/lib/pii` redaction discipline).
