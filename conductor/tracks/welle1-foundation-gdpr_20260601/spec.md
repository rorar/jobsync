## Context

JobSync is GDPR-first (Art. 5 data minimisation, accountability) and DDD-typed
(`ActionResult<T>` is the typed server-action response; `NotificationType` keys every
channel dispatch). Today `ActionResult.message` is an untyped `string` that actually
carries i18n keys, and `NotificationType` is duplicated across 13 files â€” both are
latent foot-guns. The GDPR items extend the existing `AdminAuditLog` Prisma
model/pattern (Sprint 5 Stream D) to Job-CRUD and CRM read-access, and trim the
NextAuth JWT to the minimum identifier.

## Acceptance Criteria

- [ ] **IF-5:** `ActionResult.message` is a typed i18n-key union (not bare `string`).
      All `ActionResult` producers/consumers compile against the union; build is
      zero-error. (`src/models/actionResult.ts`)
- [ ] **IF-7:** A single source-of-truth `NotificationType` union exists; the 13
      scattered definitions are removed and re-import from it. No duplicate literal sets.
- [ ] **S6a:** Every Job-CRUD mutation (create/update/delete/status-change/note) writes
      an audit-trail row (actor id, action, target id, timestamp, before/after where
      relevant) via the `AdminAuditLog` pattern. Art. 5(2) accountability.
- [ ] **S6b:** CRM read-access to `Person` PII (detail view, list-with-PII, export)
      writes a read-access audit row (who viewed which Person, when).
- [ ] **GDPR-JWT:** The NextAuth JWT carries only the user `id` â€” `email` and `name`
      are removed from the token payload. Session still resolves display fields from DB
      where needed. Art. 5(1)(c) data minimisation.
- [ ] Tests: unit for the new types + audit writers; component/integration where a
      surface changes; regression test proving the JWT no longer contains email/name.
- [ ] i18n: any new user-visible string (e.g. audit-log labels) added in en/de/fr/es.
