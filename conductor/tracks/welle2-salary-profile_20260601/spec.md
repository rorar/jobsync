# Specification: Welle 2 — Salary + Profil (Kette B)

**Track ID:** welle2-salary-profile_20260601
**Type:** Feature
**Created:** 2026-06-01
**Status:** Draft

## Summary

Add an ISO-4217 currency reference module, a user-profile address+currency form, and
structured salary (slider + currency + fixum). Currency module + profile form land
first; the Job salary migration depends on both.

## Context

JobSync already has a GeoCode Reference Module (`src/lib/connector/reference-data/
modules/geo-codes/`) exposed to the UI via the `CountrySelect` / `SubdivisionSelect`
comboboxes and the reference-data Open Host Service actions (`reference-data.actions.ts`).
This track follows that exact pattern for currency, then reuses the geo + currency
reference data in the user profile, then migrates the Job model's free-text
`salaryRange` string into a structured min/max/currency shape (reusing the existing
`format-salary-range.ts` formatter).

## User Story

As a job seeker, I want to record my home address and preferred currency on my profile
and capture each job's salary as a structured range with a currency and an optional
fixum, so that salaries display consistently and are comparable across listings.

## Acceptance Criteria

- [ ] **CUR:** A new ISO-4217 currency Reference Data module exists under
      `src/lib/connector/reference-data/` mirroring the GeoCode module (manifest, i18n,
      `index.ts` self-registration, health-only connector). It exposes currency lookup
      (code → symbol/name/minor-unit, locale-aware names) via an OHS server action.
- [ ] **F-AJ-06:** The user-profile form gains address fields (reusing `CountrySelect` +
      `SubdivisionSelect` + the reference-data OHS actions) and a currency selector
      (backed by CUR). Values persist on the Person/Profile aggregate.
- [ ] **F-AJ-05:** The Add/Edit Job form captures salary as min/max via a slider, a
      currency selector, and an optional fixum field. The `Job.salaryRange` string column
      is migrated to structured `salaryMin` / `salaryMax` / `salaryCurrency` (+ fixum).
- [ ] A Prisma migration converts existing `salaryRange` strings (best-effort parse;
      unparseable values preserved/flagged, never silently dropped).
- [ ] `format-salary-range.ts` is reused (extended, not duplicated) to render the new shape.
- [ ] Tests: unit (currency module translator + formatter), component (profile form +
      salary slider), ≥1 E2E happy-path (set profile currency; create a job with a salary range).
- [ ] i18n: all new strings + currency/field labels in en/de/fr/es.

## Dependencies

- **Internal ordering:** CUR + F-AJ-06 BEFORE F-AJ-05 (the Job form's currency selector
  reuses CUR; profile work proves the reference-data wiring).
- **Existing code:** GeoCode Reference Module pattern, `reference-data.actions.ts`
  (OHS), `CountrySelect`/`SubdivisionSelect`, `format-salary-range.ts`, `Job` Prisma model.

## Out of Scope

- Live FX/exchange-rate conversion between currencies (display in stored currency only).
- Salary analytics/aggregation dashboards.
- Crypto or non-ISO-4217 currencies.
- Editing currency reference data at runtime (it is static reference data).

## Technical Notes

- CUR is **Reference Data** (health-only connector), NOT an enrichment module — follow
  the geo-codes precedent exactly (vendored/npm data via `require()`, per-locale result
  cache). Reference lookups live in the OHS action file, never in an aggregate Repository
  (keeps the Person/Job aggregates pure — DDD).
- ADR-019: reference OHS actions are auth-gated. Validate currency codes at the boundary
  with `/^[A-Z]{3}$/`.
- Salary migration is the risky step — write the parse + migration test first, run on a
  copy, and keep a reversible mapping of original strings.
