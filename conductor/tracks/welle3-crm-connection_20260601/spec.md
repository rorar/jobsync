# Specification: Welle 3 — CRM-Verbindung (Kette C)

**Track ID:** welle3-crm-connection_20260601
**Type:** Feature
**Created:** 2026-06-01
**Status:** Draft

## Summary

Surface the existing CRM backend in the Job UI: a Person "Point of Contact" selector
inside Add Job, a Company timeline + CRM tab on the Job detail page, a recruiter-triangle
relationship, domain/pattern matching for the blocklist, and `updatedBy` FK tracking.

## Context

The CRM core (Person aggregate, `JobContact` join entity, `CrmActivityLog`,
`ActivityTimeline` component, `CrmBlocklist`) already exists in the backend. This track
is largely the **UI/wiring layer** plus two small backend extensions (recruiter triangle,
blocklist matching, updatedBy FK). It connects the Job and Person bounded contexts through
the existing `JobContact` join and the `CrmActivityLog` read model.

## User Story

As a job seeker, I want to attach a recruiter or hiring-manager contact to a job, see a
company's interaction timeline on the job page, and capture the recruiting-agency↔hiring-
company relationship, so that I have a single CRM-aware view of each application.

## Acceptance Criteria

- [ ] **F-AJ-07 / Gap-1:** Add Job includes a Person "Point of Contact" picker that
      creates a `JobContact` (optionally with role). Backend join already exists — this is
      the UI. Selecting/creating a Person reuses the Person search/create flow.
- [ ] **Gap-5:** The Job detail page gains a CRM tab embedding the existing
      `ActivityTimeline` (it already accepts props), plus a CompanyTimeline view of the
      company's `CrmActivityLog` entries.
- [ ] **F-AJ-08:** A recruiter triangle is modelled — add `recruitingCompanyId` +
      `relationshipType` so a job can distinguish the hiring company from the recruiting
      agency. Migration + UI to set it.
- [ ] **Gap-6:** `CrmBlocklist` matching supports domain + pattern matching (today only
      exact handle). Auto-creation suppression honours the new match modes.
- [ ] **Gap-7:** `updatedBy` becomes a proper FK to User (currently name-based actor only),
      on the CRM entities where actor tracking exists.
- [ ] Tests: component (contact picker, CRM tab, timeline embed), unit (blocklist matcher,
      relationship validation), ≥1 E2E happy-path (add a job with a point of contact).
- [ ] i18n: all new strings (tab labels, relationship types, picker) in en/de/fr/es.

## Dependencies

- **Internal ordering:** F-AJ-07 FIRST (establishes the contact-picker UI), then F-AJ-08
  (recruiter triangle builds on the contact concept). Gap-5, Gap-6, Gap-7 can parallelise.
- **Existing code:** `JobContact` model + `jobContact.actions.ts`, `ActivityTimeline`
  component, `CrmActivityLog` + relations, `CrmBlocklist` + `crmBlocklist.actions.ts`,
  Person search/create flow, `Company`/`Job` Prisma models.

## Out of Scope

- A standalone CRM analytics dashboard.
- Bulk import of contacts.
- Changing the Person aggregate's anonymisation/merge semantics (already shipped).
- A full company-detail page (only the timeline embed on the Job page is in scope here).

## Technical Notes

- Respect aggregate boundaries: Person/JobContact mutations go through their own action
  files (`person.actions.ts`, `jobContact.actions.ts`), never edited from `job.actions.ts`.
- ADR-015 IDOR: every CRM query includes userId; chain-traverse for sub-resources.
- F-AJ-08: `relationshipType` is an erased TS union → runtime membership check at the
  server-action boundary (ADR-019).
- Gap-6: keep the matcher ReDoS-safe; bound pattern complexity.
- Gap-7: `updatedBy` FK migration must preserve existing name-based actor strings
  (keep both, or backfill where a matching User exists) — never drop audit data.
