## Summary

`getStagedVacancies` uses skip/offset pagination, which degrades at large offsets (the DB
scans + discards `offset` rows per page). Code-verified at HEAD `b4b20e9`:
`src/actions/stagedVacancy.actions.ts:68` computes
`const offset = (safePage - 1) * safeTake` and queries `skip: offset, take: safeTake`
(~`:127`). Migrate to cursor-based pagination so deep pages stay O(page-size).

## Context

The change is not isolated to the action — it ripples through the staging UI because the
current contract is page-number based:
- `StagingContainer` (consumes the paginated result + page state),
- `RecordsPerPageSelector` (page-size control),
- `BulkActionBar` select-all (operates over the current page / "all"),
- `StagingNewItemsBanner` (new-items-since semantics interact with ordering).

The list query already uses an explicit `select` (`STAGED_VACANCY_LIST_SELECT`, M-P-02)
and orders by `discoveredAt desc` — a stable cursor key candidate, but `discoveredAt` is
not unique, so the cursor must be a composite (`discoveredAt`, `id`) to be deterministic.

(UI ripple list is code-spotted from the staging components; **verify each consumer
against the current file** before changing its props — not graph-trusted.)

## Acceptance Criteria

- [ ] `getStagedVacancies` supports cursor-based pagination (composite cursor

- [ ] Backward-compatible or cleanly-migrated contract for the four consumers
      (`StagingContainer`, `RecordsPerPageSelector`, `BulkActionBar` select-all,
      `StagingNewItemsBanner`).
- [ ] Filters (tab, status, search) + ordering still work with the cursor.
- [ ] `BulkActionBar` "select all" semantics defined under cursor pagination (select-all-
      matching-filter, not just current page) and tested.
- [ ] The new-items banner still computes correctly against the cursor ordering.
- [ ] Tests: unit (cursor query: first page, next page, last page, empty), component
      (StagingContainer paginates via cursor), regression (filters + search), perf
      sanity (deep page does not regress).

## Dependencies

## Out of Scope

- Cursor-paginating other lists (jobs, contacts) — only staging is flagged.
- Changing the staging deck (operates on a stack, not page offsets).
- Infinite-scroll UX redesign (keep the existing pagination UX unless trivially better).

## Technical Notes

- ADR-015 userId in the cursor query.
- Composite cursor is mandatory: `discoveredAt` alone is non-unique → duplicate/skip rows
  at page boundaries.
- Define "select all" precisely: under cursor pagination it should mean
  "all rows matching the current filter", resolved server-side, not "the current page".
