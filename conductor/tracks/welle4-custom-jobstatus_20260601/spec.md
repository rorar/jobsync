# Specification: Welle 4 — Custom JobStatus XL (Kette A)

**Track ID:** welle4-custom-jobstatus_20260601
**Type:** Feature
**Created:** 2026-06-01
**Status:** Draft

## Summary

Let users define their own JobStatus values + categories (XL feature), then rebuild the
status ComboBox with applied-toggle logic, reorder the Job-form layout, and drive the
Kanban columns + nav badges dynamically from the custom statuses. Allium spec FIRST.

## Context

JobSync currently has a fixed `JobStatus` set; the Kanban board, status ComboBox, and
nav badges are written against those fixed statuses. Making status user-customisable is a
domain-rule-heavy change (state machine, category semantics, Kanban column derivation,
API exposure) — exactly the kind of feature the project mandates an Allium spec for,
written BEFORE implementation. Critical ordering: **F-AJ-09 (custom status) must precede
F-AJ-02 (applied-toggle logic)** — otherwise the applied logic gets written against the
fixed statuses and has to be rewritten.

## User Story

As a job seeker, I want to define my own application statuses and group them into
categories, so that the Kanban board, status picker, and navigation badges reflect my own
workflow instead of a fixed set.

## Acceptance Criteria

- [ ] **Allium spec FIRST:** `specs/job-status.allium` captures the JobStatus state
      machine, category model, Kanban-column derivation, and API surface BEFORE code.
      It is the single source of truth; `allium:weed` reports zero spec↔code drift at end.
- [ ] **F-AJ-09:** Users can create/rename/reorder/delete JobStatus values and assign each
      to a category (XL). Statuses persist per user (ADR-015). Deleting a status in use is
      handled safely (block or reassign — defined by the spec).
- [ ] **F-AJ-02:** The status ComboBox is rebuilt with the applied-toggle logic, driven by
      the custom statuses (depends on F-AJ-09).
- [ ] **F-AJ-03:** The Job-form layout is reordered so Status sits above Date Applied.
- [ ] **Dynamic Kanban + badges:** Kanban columns and nav badges are generated from the
      user's custom statuses/categories — no hardcoded status list remains.
- [ ] A Prisma migration introduces the custom-status model and migrates existing jobs'
      fixed statuses onto seeded equivalents (no job loses its status).
- [ ] Tests: unit (state-machine/category validation, Kanban derivation), component (status
      ComboBox, Kanban columns, Job form), ≥1 E2E happy-path (create a custom status, set
      it on a job, see it as a Kanban column).
- [ ] i18n: all new strings (status management UI, category labels) in en/de/fr/es.

## Dependencies

- **Internal ordering (HARD):** Allium spec → F-AJ-09 → F-AJ-02. F-AJ-03 + dynamic
  Kanban/badges follow F-AJ-09.
- **Existing code:** `JobStatus` model + usages, the Kanban board (dnd-kit) + `useKanbanState`,
  the status ComboBox, nav badge components, `job.actions.ts`.

## Out of Scope

- Per-status automation triggers / workflow rules engine.
- Sharing custom status sets between users.
- Colour-theming beyond what category assignment needs.
- Reworking the staging deck (it operates on vacancies, not job statuses).

## Technical Notes

- Write the Allium spec with `allium:elicit` / `allium:tend`; validate with `allium:check`.
- Default-seed a category+status set on migration so existing single-user installs keep
  working zero-config; the migration maps each legacy fixed status to a seeded status.
- Kanban columns derive from categories/statuses at render — remove every hardcoded list;
  `useKanbanState` persists column order in localStorage as today.
- ADR-015 userId on every status query; deleting/reordering are mutations → ADR-019
  boundary validation for any erased union (category type).
- After the architecture lands, run `/architecture-decision-records` (custom-status model
  is an architecture decision).
