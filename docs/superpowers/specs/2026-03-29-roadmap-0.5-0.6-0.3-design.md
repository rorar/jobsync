## 2. Architectural Decisions

### AD-1: Execution Order — Event Bus First

**Decision:** Build Event Bus infrastructure (0.6 Phase 1) BEFORE completing 0.5 features.

**Rationale:** The ROADMAP states 0.6 comes after 0.5. However, 0.5's core pipeline IS done — the remaining items (Undo, Bulk Actions) all produce domain events. Building the bus first means:
- 0.5 features emit real events immediately (no "replace stubs later" debt)
- Undo/Redo can leverage events for audit trail
- `BulkActionCompleted` event is real from day one
- NotificationDispatcher can start consuming immediately

**Alternative considered:** Complete 0.5 with stubs, then build 0.6. Rejected because it creates tech debt — every `emitEvent()` call would need revisiting.

**DDD alignment:** Domain Events are first-class citizens in DDD. The bus is infrastructure that enables the domain model to communicate properly. Building it first follows the principle of establishing shared infrastructure before building features that depend on it.

### AD-2: Undo/Redo — UndoToken with Timer Expiry

**Decision:** Hybrid approach — immediate execution + reversible via UndoToken within a configurable time window (default 10s).

**Pattern:** Inspired by Saga Pattern (compensation actions) but simplified for single-process Next.js:

```
Action executed → UndoToken created (in-memory + optional DB) → Timer starts
  ├─ User clicks "Undo" or Ctrl+Z within window → Compensation action runs → Token expired
  ├─ Timer expires → Token removed → Action becomes permanent
  └─ Batch: one UndoToken per batch (not per item)
```

**Why not full Command Pattern?** Over-engineered for a self-hosted app. The Command Pattern with replay would require event sourcing infrastructure. The UndoToken approach achieves the same UX (toast undo, Ctrl+Z) with 10% of the complexity.

**Why not pure timer-delay?** Delaying execution creates UX problems (user expects immediate feedback). Gmail's "Undo Send" pattern (execute immediately, offer reversal) is the proven approach.

**Implementation:**
- `UndoStore` — in-memory Map<tokenId, UndoEntry> with TTL
- `UndoEntry` — { action, compensationFn, itemIds, expiresAt }
- `useUndoStack()` hook — manages Ctrl+Z keybinding, exposes `undo(tokenId)`
- Toast component shows "Undo" button with countdown timer

### AD-3: Event Bus — In-Process TypedEventBus

**Decision:** In-process EventEmitter with typed event registry. No external dependencies.

**Pattern:** Application Service (not infrastructure service) — the bus lives in the same process as the Next.js server.

```typescript
// src/lib/events/event-bus.ts
interface EventBus {
  publish<T extends DomainEvent>(event: T): void;
  subscribe<T extends DomainEvent>(type: T['type'], handler: EventHandler<T>): Unsubscribe;
}
```

**Why not Redis/NATS/Kafka?** Self-hosted app, single process. External message brokers add operational complexity for zero benefit at this scale.

**Future-proofing:** The `EventBus` interface is abstract enough to swap to Redis Pub/Sub later (ROADMAP 7.1 Public API could need cross-process events). Consumers don't know or care about the transport.

**Consumers at launch:**
1. `NotificationDispatcher` — creates In-App notifications
2. `AuditLogger` (optional) — logs events for debugging
3. Future: CRM (5), Data Enrichment (1.13), Document Generation (4.2)

### AD-4: Bulk Actions — Domain Service with Partial-Success Semantics

**Decision:** `BulkActionService` as a Domain Service, not just a UI loop.

**Pattern:** Fan-Out/Fan-In from workflow orchestration:
1. Validate all items upfront (fail-fast for obviously invalid)
2. Execute each item individually (catch per-item errors)
3. Aggregate results into `BulkActionResult`
4. Emit single `BulkActionCompleted` event
5. Create single UndoToken for the batch

**Partial-Success:** Invalid items are skipped, not rolled back. The response tells the user which succeeded and which failed.

### AD-5: Dedup-Retention — Scheduled Cleanup with Privacy by Design

**Decision:** Background cleanup job (triggered by cron or manual) that:
1. Finds StagedVacancies past retention period
2. Extracts dedup hash before deletion
3. Inserts into DedupHash table
4. Deletes StagedVacancy data
5. Runner checks DedupHash before inserting new vacancies

**DSGVO alignment:** Hash is one-way (SHA-256), not reconstructible. Minimal data retention: hash + userId + timestamp.

### AD-6: Notification Preferences — JSON Column on UserSettings

**Decision:** Add `notificationPreferences` JSON column to existing `UserSettings` model.

**Schema:**
```typescript
interface NotificationPreferences {
  enabled: boolean;                          // global kill switch
  channels: { inApp: boolean; /* future: email, push, webhook */ };
  perType: Record<NotificationType, { enabled: boolean; channel?: string }>;
  quietHours?: { start: string; end: string; timezone: string };
  digestMode?: 'immediate' | 'hourly' | 'daily';
}
```

**Why JSON, not normalized tables?** Preferences are a single document owned by one user. No cross-user queries needed. JSON is simpler and aligns with existing `UserSettings` pattern.

### AD-7: 0.3 Follow-Ups — Minimal Scope

**`?:` vs `| null` unification:** Apply the project invariant (`null` in DB = `| null` in model, `undefined` = "field not in response") to `task.model.ts` and any other models with `?:` for DB-nullable fields.

**Mapper functions:** Create `toTask()` and `toProfile()` mapper functions following the same pattern as `toStagedVacancy()` — narrow Prisma strings to domain enums.

### DDD Alignment
- **Event Bus** = Shared Kernel (infrastructure used by all bounded contexts)
- **NotificationDispatcher** = Application Service (subscribes to domain events, routes to channels)
- **BulkActionService** = Domain Service (orchestrates multi-entity operations)
- **UndoStore** = Infrastructure Service (manages compensation tokens)
- **RetentionService** = Domain Service (enforces DSGVO lifecycle rules)

## 5. Allium Spec Requirements

### specs/event-bus.allium
- Entity: DomainEvent (type, timestamp, payload)
- Entity: EventSubscription (eventType, handler, priority)
- Contract: EventBus (publish, subscribe, unsubscribe)
- Rule: ErrorIsolation (one consumer failure doesn't affect others)
- Rule: OrderGuarantee (events delivered in publish order per type)

### specs/notification-dispatch.allium
- Entity: Notification (id, userId, type, message, read, createdAt)
- Entity: NotificationPreference (userId, type, enabled, channel)
- Contract: NotificationDispatcher (dispatch, checkPreferences)
- Rule: QuietHours (suppress during configured hours)
- Rule: Deduplication (no duplicate notifications within window)
- Rule: ChannelRouting (preference → channel selection)
- Surface: NotificationBell (unread count, dropdown)

### specs/vacancy-pipeline.allium (extend existing)
- Entity: UndoToken (id, action, compensation, itemIds, expiresAt)
- Rule: UndoExpiry (token invalid after TTL)
- Rule: BulkPartialSuccess (invalid items skipped, not rolled back)
- Rule: BatchUndoGranularity (one undo per batch)
- Rule: RetentionCleanup (delete data, preserve hash)
- Surface: BulkActionBar (multi-select, operations)
- Surface: UndoToast (countdown, undo button)

## 6. Risk Assessment

| Risk | Mitigation |
|------|-----------|
| In-memory UndoStore lost on server restart | Acceptable for 10s window. Document that undo only works within window. |
| EventBus error in consumer blocks other consumers | Error isolation: try/catch per consumer, log failures |
| Bulk promote opens N dialogs | Batch PromotionDialog with shared defaults |
| Notification spam from automation runs | Batch summary notification (e.g., "12 new vacancies discovered") not per-item |
| Retention cleanup on large datasets | Process in batches of 100, yield between batches |
| Ctrl+Z conflicts with browser undo | Only capture when focus is on app container, not input fields |
